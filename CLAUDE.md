# CLAUDE.md — ذاكرة مشروع دكانجي

**قاعدة إلزامية لأي جلسة عمل (بشرية أو ذكاء اصطناعي):**
1. اقرأ هذا الملف كاملاً قبل البدء بأي مهمة — خصوصاً قسم "الأخطاء المتكررة المعروفة" وقسم "سجل الإصلاحات" في الأسفل، للتأكد أن ما تحاول حلّه لم يُحلّ سابقاً بطريقة معينة يجب اتّباعها أو تجنّب تكرارها.
2. لا تُغلق أي مهمة (إصلاح أو ميزة جديدة) قبل إضافة سطر جديد في "سجل الإصلاحات" أسفل هذا الملف. هذا الملف هو الذاكرة الوحيدة الدائمة للمشروع بين الجلسات المختلفة.

---

## 1. نظرة عامة سريعة

- **المنصة:** دكانجي — سوق طلبات محلي (مطاعم وسوبرماركت) في تركيا.
- **الواجهة الأمامية:** `index.html` + ملف واحد `app.js` (~12,884 سطراً، 684 دالة) + `styles.css`.
- **الخلفية:** Vercel serverless functions تحت `api/` + قاعدة بيانات وتخزين ملفات على Supabase.
- **التطبيقات الجوالة:** عبر Capacitor (`android/`, `ios/`).
- **Service Worker:** `sw.js` — تخزين مؤقت + إشعارات Push.

## 2. مصدر الحقيقة الوحيد للبيانات — مهم جداً

- بيانات المتاجر والمنتجات الفعلية تُقرأ **حصرياً من Supabase**. مجلد `migrations/` يوثّق كل تغيير على المخطط بالترتيب الزمني (اسم الملف).
- ملفات `*-data.js` في جذر المشروع (56 ملفاً، مثل `zaitoune-data.js`) هي **كود قديم ميت (dead code)** — تأكدنا أن لا شيء في `app.js` أو `server.js` يستدعيها حالياً. **لا تُعدَّل ولا تُعتمد كمصدر بيانات أبداً.**
- عند التعامل معها نهائياً: انقلها إلى مجلد `archive/` أو احذفها، ونظّف مصفوفة `APP_SHELL` في `sw.js` من كل إشارة إليها (لا تزال تحتوي حالياً على ~30 إشارة قديمة لها).
- أي خطأ متعلق بظهور/اختفاء متجر → تحقّق أولاً من سجلّه في Supabase، وليس من هذه الملفات القديمة.

## 3. قواعد العمل بـ Git (إلزامية)

1. لا تترك أي فرع (branch) أو نسخة عمل (`git worktree`) مفتوحة لأكثر من يوم عمل واحد. كل ميزة/إصلاح: فرع قصير العمر → commit → دمج في `main` فوراً → حذف الفرع محلياً وبعيداً.
2. قبل البدء بأي عمل جديد:
   ```
   git checkout main
   git pull
   git status   # يجب أن يكون نظيفاً؛ إن لم يكن، توقف واسأل قبل المتابعة
   ```
3. عند الانتهاء من أي مهمة:
   ```
   git worktree remove <path>
   git branch -d <branch-name>
   git push origin --delete <branch-name>
   ```
4. لا تدمج فرعاً عمره أكثر من أسبوعين دون مراجعة يدوية كاملة لمحتواه — قد يعيد كوداً تجاوزه الزمن ويُلغي إصلاحات لاحقة.
5. إذا ظهر خطأ "Unable to unlink .git/index.lock"، فهذا يعني وجود عملية git عالقة أو متزامنة؛ تحقق من ذلك قبل حذف القفل يدوياً.
6. **ملاحظة حالة معروفة (2026-07-10):** المشروع كان يعمل على فرع `supermarket-baseline-categories` (وليس `main`) مع 111 ملفاً معدَّلاً وغير محفوظ. راجع حالة هذا الفرع قبل بدء أي عمل جديد ولا تتراكم عليه المزيد من التعديلات غير المحفوظة.
7. **تنظيف دوري:** الجهاز يستخدم PowerShell (لا يوجد bash افتراضياً)، لذا شغّل `.\scripts\git-cleanup.ps1` من PowerShell في جذر المشروع كل أسبوع تقريباً لحذف نُسخ العمل والفروع المدموجة تلقائياً (يوجد أيضاً `scripts/git-cleanup.sh` لمن يفضّل Git Bash). راجع يدوياً القائمة التي يطبعها السكربت في النهاية قبل حذف أي فرع منها.

## 4. قائمة تحقق النشر الإلزامية (Deploy Checklist)

عند تعديل `app.js` أو `styles.css`، رقم الإصدار `?v=NNN` يجب تحديثه معاً في **3 مواضع**:

1. `index.html` → وسم `<script src="/app.js?v=NNN">` ووسم `<link href="/styles.css?v=NNN">`.
2. `sw.js` → الثابت `CACHE = "dukkanci-vNNN"`.
3. `sw.js` → كل إشارة لـ `app.js?v=` و `styles.css?v=` داخل مصفوفة `APP_SHELL`.

**قبل كل نشر، شغّل:**
```
npm run check:versions
```
هذا السكربت (`scripts/check-versions.js`) يفشل صراحةً إن كانت الأرقام غير متطابقة بين `index.html` و`sw.js`، بدل اكتشاف الخطأ بعد وصوله للمستخدمين.

> **حالة معروفة عند كتابة هذا الملف:** `index.html` على `app.js?v=264` / `styles.css?v=161`، بينما `sw.js` لا يزال على `app.js?v=176` / `styles.css?v=111`. يجب تصحيح هذا الانحراف في أقرب نشر قادم.

## 5. خريطة الملفات الأساسية

| المسار | الدور |
|---|---|
| `index.html` | الصفحة الرئيسية وكل وسوم `<script>`/`<link>` |
| `app.js` | كل منطق الواجهة (عربة، دفع، عرض متاجر، لوحة تاجر، تتبع...) — ملف ضخم، **ابحث عن الدالة بالاسم قبل إضافة أي منطق جديد** لتجنّب التكرار |
| `styles.css` | كل الأنماط |
| `sw.js` | Service Worker: تخزين مؤقت + إشعارات Push |
| `api/` | دوال Vercel serverless (طلبات، دفع، خرائط، حملات...) |
| `lib/` | أدوات مشتركة (بوابة AI، معالجة صور الكتالوج، Google indexing...) |
| `migrations/` | كل تغييرات مخطط Supabase، مرتبة بالتاريخ في اسم الملف |
| `scripts/` | سكربتات بناء/استخراج بيانات لمرة واحدة لكل متجر — **ليست** جزءاً من التشغيل الحي |
| `catalog/` | الكتالوج المشترك بين متاجر السوبرماركت |
| `*-data.js` (جذر المشروع) | **كود قديم ميت — لا تستخدمه** (راجع قسم 2) |

## 6. الأخطاء المتكررة المعروفة (راجعها قبل إعادة الاختراع)

- **تجمد زر إرسال الطلب / رسوم توصيل صفرية** للمتاجر بدون إحداثيات — أُصلح منطقياً في `app.js`، لكن ظهور الإصلاح فعلياً يتطلب تطبيق قسم 4 (رفع رقم الإصدار في الأماكن الثلاثة) وإلا يبدو "غير مطبَّق" رغم أن الكود صحيح.
- **ظهور/اختفاء متجر** بعد إضافته مباشرة إلى Supabase — تأكد أن منطق العرض في `app.js` يقرأ نفس الحقول التي يُدخلها Supabase، وليس من `*-data.js`.
- **فقدان طلبات صامت** — عولج جزئياً عبر مسار حفظ مستقل في `api/notify-order.js` بدل الاعتماد فقط على كتابة المتصفح المباشرة. أي تعديل مستقبلي على تدفق الطلب يجب أن يمر عبر هذا المسار أيضاً وليس فقط الكتابة المباشرة.
- **ثغرات API غير محمية** — تحقّق دائماً من التحقق من الهوية/الصلاحيات عند إضافة أي `endpoint` جديد داخل `api/`.
- **حذف متجر من Supabase لا يكفي وحده** — كل متاجر السوبرماركت/المطاعم القديمة مبنية كملف `<slug>-data.js` مُحمَّل عبر `<script>` مباشر في `index.html` ومُدرَج أيضاً في `sw.js` (مصفوفة `APP_SHELL`)، وليس مجرد كود ميت كما هو الحال مع الـ56 ملف `*-data.js` المذكورة في قسم 2. حذف صف المتجر من Supabase فقط (بدون حذف `<slug>-data.js` وإزالة أسطره الثلاثة من `app.js`: `stores.push(...)`, `products.push(...)`, `...XDeliverySettings`) يترك المتجر ظاهراً على الموقع فعلياً رغم اختفائه من قاعدة البيانات — هذا بالضبط ما حدث مع متجر حلاب 1881 (id 49) بعد إزالته من Supabase في 2026-07-10 دون تنظيف الحزمة الأمامية.
