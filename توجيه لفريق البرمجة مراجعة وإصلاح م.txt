توجيه لفريق البرمجة: مراجعة وإصلاح مشروع دكانجي
المطلوب من الفريق إجراء مراجعة تقنية كاملة وإصلاح جميع المشاكل الحرجة قبل اعتبار المشروع جاهزًا للإطلاق. المشروع حاليًا يصلح كنسخة عرض/MVP، لكنه غير آمن للإنتاج قبل معالجة النقاط التالية.
1. إغلاق مشاكل الأمان والصلاحيات — أولوية قصوى
يمنع إطلاق المشروع قبل تنفيذ التالي:
مراجعة سياسات Supabase الحالية.
إزالة جميع سياسات for all using (true) الموجودة في جداول:stores
products
orders
complaints
integration_settings

استبدالها بسياسات RLS حقيقية:العميل يرى طلباته فقط.
التاجر يرى ويعدل متجره ومنتجاته وطلباته فقط.
الإدارة فقط تستطيع رؤية كل شيء.

عدم السماح للـ anon key بكتابة أو تعديل بيانات حساسة.
التأكد أن service_role لا يستخدم أبدًا في الواجهة.
معيار القبول: لا يستطيع مستخدم غير مصرح له تعديل متجر، منتج، طلب، شكوى، أو إعدادات تكاملات من المتصفح أو API.
2. إصلاح دخول التاجر
الدخول الحالي برقم واتساب فقط غير مقبول، لأن رقم المتجر ظاهر للعموم.
المطلوب:
إزالة تسجيل الدخول برقم الهاتف فقط.
اعتماد Supabase Auth:WhatsApp OTP أو SMS OTP.
أو Google Login.

إنشاء علاقة واضحة بين المستخدم والمتجر، مثل جدول:store_users
user_id
store_id
role

لا يسمح لأي تاجر بإدارة متجر إلا إذا كان مرتبطًا به رسميًا.
طلبات إنشاء متجر جديد لا تنشر مباشرة، بل تدخل حالة “بانتظار موافقة الإدارة”.
معيار القبول: معرفة رقم واتساب المتجر لا تكفي للدخول إلى لوحة المتجر.
3. إصلاح لوحة الإدارة
المطلوب:
عدم إرسال كلمة مرور الإدارة في query string نهائيًا.
إزالة استخدام ?key=password.
استخدام جلسة آمنة أو Supabase Auth مع role admin.
عدم حفظ كلمة مرور الإدارة كنص مباشر في sessionStorage.
تفعيل صلاحيات الإدارة عبر backend فقط.
حماية كل عمليات الإدارة من جهة الخادم، وليس من الواجهة فقط.
معيار القبول: لا تظهر كلمة مرور الإدارة في الرابط، ولا في logs، ولا يمكن استدعاء API الإدارة دون صلاحية حقيقية.
4. إزالة ملف كلمات المرور
يوجد ملف باسم:
dukkanci project passwords.txt
المطلوب:
نقله خارج مجلد المشروع فورًا.
التأكد أنه غير مرفوع إلى Git.
إضافته إلى .gitignore إن لزم.
تدوير/تغيير أي مفاتيح أو كلمات مرور كانت داخله إذا كان قد تم مشاركته أو رفعه سابقًا.
عدم فتح أو طباعة محتواه في logs أو تقارير.
معيار القبول: لا توجد أي أسرار أو كلمات مرور داخل المشروع.
5. حماية WhatsApp API
المطلوب مراجعة api/notify-order.js بالكامل.
يجب تنفيذ التالي:
التحقق من توقيع Meta Webhook، وليس قبول أي body يدعي أنه WhatsApp.
جعل NOTIFY_SECRET إلزاميًا لأي webhook داخلي من Supabase أو النظام.
حماية تحديث حالة الطلب بصلاحيات التاجر/الإدارة.
عدم السماح لأي مستخدم بإرسال إشعار حالة لأي رقم عميل.
فصل مسؤوليات endpoint إن أمكن:webhook استقبال WhatsApp.
إشعارات الطلبات.
صندوق محادثات الإدارة.
ردود الإدارة.

التحقق من rate limit لمنع الإساءة.
معيار القبول: لا يمكن لطرف خارجي إرسال رسائل WhatsApp أو تغيير حالة طلب عبر API مفتوح.
6. معالجة مخاطر XSS
هناك أماكن كثيرة في app.js يتم فيها إدخال بيانات المتجر/المنتج مباشرة داخل HTML.
المطلوب:
استخدام escape/sanitize لكل بيانات قادمة من قاعدة البيانات أو إدخال التاجر:اسم المنتج.
وصف المنتج.
اسم المتجر.
وصف المتجر.
العنوان.
التصنيفات.
النصوص الإدارية.

منع إدخال HTML من لوحة التاجر أو الإدارة إلا إذا كان مصفى بشكل صارم.
مراجعة كل استخدامات innerHTML.
أي قيمة ديناميكية داخل HTML يجب أن تمر عبر دالة escape موحدة.
معيار القبول: إدخال نص مثل <script>alert(1)</script> في اسم منتج لا ينفذ ككود ولا يكسر الصفحة.
7. إصلاح الخادم المحلي server.js
الخادم المحلي يدعم فقط /api/delivery-quote، بينما الواجهة تستدعي endpoints أخرى مثل:
/api/config
/api/reverse-geocode
المطلوب:
إما جعل server.js يشغل/يمرر كل ملفات /api/*.
أو توثيق أن التطوير المحلي يتم عبر Vercel dev.
توحيد منطق delivery-quote بين server.js وapi/delivery-quote.js.
حاليًا الحساب المحلي لا يطبق نفس التقريب والحد الأدنى الموجود في API الإنتاجي.
معيار القبول: نفس الطلب يعطي نفس رسوم التوصيل محليًا وإنتاجيًا.
8. إصلاح Service Worker وPWA cache
يوجد عدم تطابق بين نسخ الملفات:
index.html يستخدم app.js?v=97
sw.js يستخدم app.js?v=89
styles.css كذلك غير متطابق.
المطلوب:
تحديث sw.js.
توحيد أرقام cache.
التأكد أن المستخدم يحصل على النسخة الجديدة بعد النشر.
عدم caching ملفات API.
اختبار offline mode بعد التحديث.
معيار القبول: بعد نشر نسخة جديدة لا يبقى المستخدم عالقًا على كود قديم.
9. مراجعة تدفق إنشاء المتجر
المتطلبات تنص أن قبول المتجر يتم بعد مراجعة الإدارة، لكن الكود ينشئ المتجر مباشرة.
المطلوب:
إنشاء جدول merchant_applications أو حالة approval_status.
الحالات المقترحة:pending
approved
rejected
suspended

المتجر لا يظهر للعملاء قبل الموافقة.
الإدارة تستطيع قبول/رفض/تعليق المتجر.
التاجر يستطيع تعديل بياناته، لكن بعض التعديلات الحساسة قد تحتاج مراجعة.
معيار القبول: أي طلب انضمام جديد لا يظهر في الموقع قبل موافقة الإدارة.
10. استكمال نظام الطلبات
المطلوب:
دعم حالات الطلب كما في المتطلبات:طلب جديد
بانتظار تأكيد المتجر
تم القبول
يوجد تعديل مطلوب
بانتظار موافقة العميل
بانتظار الدفع
قيد التجهيز
جاهز للاستلام
خرج للتوصيل
مكتمل
ملغى

السماح للتاجر قبل التأكيد بـ:حذف منتج.
تعديل كمية.
اقتراح بديل.
رفض الطلب.
تعديل السعر النهائي مع سبب.

العميل يجب أن يوافق على النسخة المعدلة قبل الدفع.
الدفع بالبطاقة لا يظهر كتدفق حقيقي قبل ربط iyzico.
معيار القبول: الطلب يمكن تعديله وموافقته وتتبع حالته من العميل والتاجر والإدارة.
11. الدفع والاشتراكات
حاليًا iyzico مذكور في الواجهة فقط.
المطلوب:
إما إخفاء خيار الدفع بالبطاقة مؤقتًا.
أو ربط iyzico فعليًا.
ربط اشتراكات المتاجر بنظام حقيقي:تاريخ بداية.
تاريخ نهاية.
حالة الاشتراك.
خطة الاشتراك.
طريقة الدفع.

عند انتهاء الاشتراك يجب تطبيق السياسة المطلوبة:المتجر يظهر.
لكنه لا يستقبل طلبات جديدة.

معيار القبول: لا يوجد زر أو خيار يوهم المستخدم بوجود دفع إلكتروني غير منفذ.
12. تحسين الأداء
المشروع يحمل بيانات كثيرة جدًا دفعة واحدة.
المطلوب:
عدم تحميل كل ملفات بيانات المنتجات في أول زيارة.
تحميل المنتجات حسب:المتجر.
البحث.
التصنيف.

ضغط الصور الكبيرة.
تحويل الصور الضخمة إلى WebP/AVIF.
مراجعة الصور التي تتجاوز 1MB.
إضافة lazy loading حقيقي.
دراسة تقسيم app.js الكبير إلى وحدات.
معيار القبول: الصفحة الرئيسية تفتح بسرعة مقبولة على موبايل متوسط واتصال 4G.
13. إصلاح SEO وSSR
ملفات SSR تستخدم نطاقًا ثابتًا:
https://dukkanci.vercel.app
بينما canonical يستخدم:
https://www.dukkanci.com.tr
المطلوب:
عدم الاعتماد على نطاق قديم لجلب shell.
استخدام origin الحالي أو متغير بيئة واضح.
التأكد أن صفحات:/store/:slug
/product/:slug
/category/:slug
ترجع HTML صحيحًا، status صحيحًا، canonical صحيحًا.

اختبار 404 للمنتجات غير الموجودة.
معيار القبول: لا يوجد اعتماد هش على دومين خارجي قديم.

تصدير:العملاء.

15. الاختبارات المطلوبة قبل التسليم
يجب إضافة أو تنفيذ اختبارات يدوية موثقة على الأقل للتالي:
مستخدم عادي لا يستطيع تعديل منتجات.
تاجر لا يستطيع إدارة متجر غير متجره.
تاجر لا يرى طلبات متجر آخر.
الإدارة فقط ترى كل الطلبات.
إنشاء متجر جديد يبقى pending.
XSS لا يعمل في أسماء المنتجات أو المتاجر.
Checkout يعمل.
حساب التوصيل يعطي نفس النتيجة محليًا وإنتاجيًا.
WhatsApp API لا يقبل طلبات غير موقعة/غير مصرح بها.
Service Worker يحدث الملفات بعد نشر جديد.
ملاحظة ختامية للفريق
لا تبدأوا بإضافة ميزات جديدة قبل إغلاق مشاكل الأمن والصلاحيات. الأولوية ليست تجميل الواجهة الآن، بل تحويل المشروع من “واجهة MVP” إلى “نظام إنتاجي آمن”.
ترتيب العمل المقترح:
الأمن وRLS.
Auth وصلاحيات التاجر/الإدارة.
حماية APIs.
إصلاح XSS.
إصلاح server.js وService Worker.
تدفقات الطلب والدفع.
الأداء والـ SEO.
الميزات الثانوية.
